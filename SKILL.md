# 旬彩坊 週次取引集計 SKILL.md
**バージョン: 2.1（2026-04-13 aggregate.py v2.0に合わせて修正）**

---

## 概要

合同会社旬彩坊（野菜仲卸）の週次売上を、4つのシステムから統合集計するためのルール集。  
毎週月曜に前週（月〜土）の集計を実行する。

---

## ディレクトリ構成

```
旬彩坊_週次集計/
├── SKILL.md                          # 本ファイル
├── input/
│   └── {YYYY-WNN}/                   # 例: 2026-W14
│       ├── インフォマート.csv
│       ├── タノム_{YYYY-WNN}_-_delivery.csv   # ※「納品書一覧」のみ
│       ├── アスピット.csv
│       └── 販売大臣_{YYYY-WNN}.xlsx
├── master/
│   └── 取引先マスタ.csv
└── output/
    └── weekly_summary.xlsx
```

**週番号の定義**: 納品日ベース。月曜〜日曜の週。  
例: 2026-W14 = 2026-03-30（月）〜 2026-04-05（日）

---

## 業務上の基本知識

| 項目 | 内容 |
|------|------|
| 配送休業日 | **水曜・日曜は配送なし**。この2日のデータが欠落していても異常ではない |
| 金額0円の商品 | **異常ではない**。魚・野菜の相場品は集計時点で単価未定のまま納品される通常運用 |
| 仮商品（その他野菜 等） | 定番にない品をコメント欄で追加注文する商習慣。注文一覧には現れず、**納品書一覧にのみ現れる** |
| 会社と店舗の関係 | 1社が複数店舗を持つケースが多い。将来的に会社単位の集計も必要 |
| マイナス行 | 返品処理。**そのまま集計に含める**（除外しない） |

## データソース仕様

### ソース1: インフォマート

| 項目 | 内容 |
|------|------|
| ファイル名 | `インフォマート.csv` |
| 文字コード | **Shift-JIS（cp932）** |
| 区切り | カンマ区切り |

**ヘッダー構造（必読）**
- 1行目: `"H","日付",...` → **スキップする**
- 2行目: 列名行。列名は全角カッコ付き（例: `［伝票日付］`）
- 3行目以降: データ行

**集計対象行**
- `区分` 列が `D` の行のみ集計する
- `区分 = F`（フッター合計行）は**必ず除外する**

**使用する列**

| 用途 | 列名 | 注意 |
|------|------|------|
| 店舗名 | `［取引先名］` | ⚠️ `［納品場所名］` は使ってはいけない（`-既定納品場所-` という無意味な値が入る） |
| 納品日 | `［伝票日付］` | |
| 金額 | `［小計］` | そのまま使える（再計算不要） |

**法人略称の抽出ルール**
- 店舗名の全角カッコ内に法人略称が入る
- 例: `ＦＣ　ＣＫ本厚木（ＫＷＤ）` → 略称は `KWD`
- カッコがない店舗名もある（その場合は略称なしとして扱う）

**想定規模（参考）**
- 週次合計: 約376万円 / 約100店舗

---

### ソース2: タノム

| 項目 | 内容 |
|------|------|
| ファイル名 | `タノム_{YYYY-WNN}_-_delivery.csv`（納品書一覧） |
| 文字コード | **UTF-8（BOMなし）** |
| 区切り | カンマ区切り |

⚠️ **「注文一覧」は絶対に使わない**  
注文一覧は小計列が伝票合計の繰り返しになっており使えない。また`伝票日付`は発注日であり納品日ではない。  
**必ず「納品書一覧」（ファイル名に `_delivery` が含まれるもの）を使う。**

**ヘッダー構造**
- 1行目が列名。全角カッコなしの通常の列名

**使用する列**

| 用途 | 列名 | 注意 |
|------|------|------|
| 店舗名 | `取引先名` | |
| 納品日 | `納品日` | |
| 商品名 | `明細商品名` | |
| 単価 | `明細単価` | |
| 数量 | `明細数量` | |
| 金額 | `明細単価 × 明細数量` | ⚠️ `小計`列は使わない。必ず単価×数量で再計算する |

**日付絞り込み**
- CSVダウンロード時に既に納品日で絞り込まれているため、**スクリプト側での日付フィルタリングは不要**

**新旧フォーマットの自動判別**
- `納品番号` 列が存在する → 新形式（納品書一覧）として処理
- `納品番号` 列がない → 旧形式（`［データ区分］`スタイル）として処理
- スクリプトが自動判別するため、手動での切り替えは不要

**注文一覧と納品書一覧の差が生じる理由（参考知識）**
- キャンセル・返品・数量変更・単価確定後の変更
- コメント欄で追加発注された定番外商品（「その他野菜」等）

**想定規模（参考）**
- 週次合計: 約239万円 / 46店舗

---

### ソース3: アスピット

| 項目 | 内容 |
|------|------|
| ファイル名 | `アスピット.csv` |
| 文字コード | **Shift-JIS（cp932）** |
| 区切り | カンマ区切り |

**特徴**
- 全件が1社「**株式会社アイティープラス**」傘下の10店舗のみ
- 注文ベースのデータ（納品確定後のデータ取得方法が現状不明）
- 全体の約6%を占めるため、注文ベースのまま許容

**使用する列**

| 用途 | 列名 | 注意 |
|------|------|------|
| 店舗名 | `店舗名称` | |
| 会社名 | `発注企業名` | ◎ 会社情報が明示されている唯一のソース |
| 納品日 | `納品日` | ⚠️ `発注日` は空欄が多いため使わない |
| 金額 | `原価金額` | |

**除外する列**
- 末尾の `Unnamed: 25` 列は無視する（空列）

**想定規模（参考）**
- 週次合計: 約78万円 / 10店舗

---

### ソース4: 販売大臣

| 項目 | 内容 |
|------|------|
| ファイル名 | `販売大臣_{YYYY-WNN}.xlsx` |
| ファイル形式 | **.xlsx**（元は .XLS だが変換済みのものを使う） |

**⚠️ ファイル準備の運用フロー（毎週必須）**  
元ファイルは `.XLS` 形式のため、そのままではCoworkが読めない。  
1. `.XLS` ファイルをGoogleドライブにアップロード  
2. Googleスプレッドシートで開く  
3. `ファイル → ダウンロード → Microsoft Excel (.xlsx)` で保存  
4. 保存したファイルを `input/{週}/` フォルダに置く  
ファイル名は必ず `販売大臣_{YYYY-WNN}.xlsx` に統一する。

**集計対象行**

| 種コード | 意味 | 集計 |
|----------|------|------|
| `1` | 明細行 | ✅ 集計する |
| `3` | 伝票計 | ❌ 除外 |
| `18` | 税計 | ❌ 除外 |

**`種` 列が `1` の行のみを集計対象とする。**

**使用する列**

| 用途 | 列名 | 注意 |
|------|------|------|
| 店舗名 | `得意先` | 店舗と会社が混在している可能性あり（要確認） |
| 得意先コード | `得意先コード` | 名寄せ・マスタ整備に活用 |
| 納品日 | `伝票日付[年]` + `伝票日付[月]` + `伝票日付[日]` | **3列を結合して日付を作る** |
| 金額 | `金額` | ※ `粗利` 列も存在（他ソースにはない情報） |

**日付結合の方法**
```python
df['納品日'] = pd.to_datetime(
    df['伝票日付[年]'].astype(str) + '-' +
    df['伝票日付[月]'].astype(str).str.zfill(2) + '-' +
    df['伝票日付[日]'].astype(str).str.zfill(2)
)
```

**特徴**
- FAX・電話・LINEなどプラットフォーム非経由の直取引顧客
- 1店舗あたり単価が他チャネルより高い（約9.5万円/週）
- `粗利` 列がある（将来的に粗利分析に活用できる）

**想定規模（参考）**
- 週次合計: 約455万円 / 48店舗

## 集計処理のフロー

```
1. 各CSVの読み込み（文字コード・ヘッダー処理に注意）
       ↓
2. 不要行の除外（インフォマート: 区分=F / 販売大臣: 種≠1）
       ↓
3. 使用列の統一（店舗名・納品日・金額）
       ↓
4. ソース列の付与（どのシステムから来たデータか記録）
       ↓
5. 4ソースを縦方向に結合
       ↓
6. 日付フィルタリング（対象週の月〜土のみ）※ タノムは不要
       ↓
7. 店舗別・日別に集計
       ↓
8. output/weekly_summary.xlsx に出力
```

## 統一後のカラム設計（結合後のデータフレーム）

| カラム名 | 型 | 内容 |
|----------|-----|------|
| `source` | str | `インフォマート` / `タノム` / `アスピット` / `販売大臣` |
| `店舗名` | str | 各ソースの店舗名（正規化前の生値） |
| `納品日` | date | 納品日（yyyy-mm-dd） |
| `金額` | int | 円（マイナス含む） |
| `得意先コード` | str | 販売大臣のみ。他はNaN |
| `粗利` | int | 販売大臣のみ。他はNaN |

## 出力仕様（weekly_summary.xlsx）

**週次追記モード**: 同じファイルに毎週のデータを追記していく。週をまたいだ比較が可能。

### シート1: 店舗別週次サマリ

| 列 | 内容 |
|----|------|
| 集計週 | 例: 2026-W14 |
| ソース | インフォマート／タノム／アスピット／販売大臣 |
| 店舗名 | 正規化後の店舗名 |
| 法人略称 | インフォマートのカッコ内略称 |
| 取引件数 | 明細行数 |
| 合計金額（円） | 週次合計 |

### シート2: 全明細

| 列 | 内容 |
|----|------|
| 集計週 | |
| ソース | |
| 取引日 | yyyy-mm-dd |
| 店舗名 | |
| 法人略称 | |
| 商品名 | |
| 金額 | 円（マイナス含む） |

### シート3: 異常検知

異常種別ごとに記録。以下の4種類を検知する。

| 異常種別 | 条件 |
|----------|------|
| 金額0円 | 金額が0の明細（異常ではないが記録） |
| 金額マイナス（返品） | 金額がマイナスの明細 |
| 重複明細 | 同一ソース・取引日・店舗名・商品名が重複 |
| 前週比±50%超 | 前週データがある店舗で50%以上の増減 |

### シート4: マスタ未ヒット店舗

取引先マスタに登録されていない店舗の一覧。マスタ整備の作業リストとして活用する。

## 取引先マスタ（master/取引先マスタ.csv）

**用途**: 表記ゆれの名寄せ。aggregate.pyが読み込み、`表記ゆれ`列の値を`正式名称`に統一する。

**カラム設計（ヘッダー）**
```
正式名称,表記ゆれ,除外,除外理由,備考
```

| カラム | 内容 |
|--------|------|
| 正式名称 | 正規化後の統一店舗名 |
| 表記ゆれ | パイプ（`|`）区切りで複数指定可。例: `ものみの塔|ものみの塔（フジタカ連帯）` |
| 除外 | `1` を入れると週次集計から除外される。空欄は集計対象 |
| 除外理由 | 除外する理由のメモ（任意） |
| 備考 | その他の特記事項（任意） |

**除外対象（現在登録済み）**

| 店舗名 | 除外理由 |
|--------|----------|
| レンブラントホテル海老名 | 月次まとめ請求のため週次集計が歪む（経理確認済み） |
| うおバルアフロ | 月次まとめ請求のため週次集計が歪む（経理確認済み） |

除外された店舗は集計には含めないが、**シート5「除外明細」に別途記録する**。

**記載例**
```csv
正式名称,表記ゆれ,除外,除外理由,備考
レンブラントホテル海老名,,1,月次まとめ請求,経理確認済み
うおバルアフロ,,1,月次まとめ請求,経理確認済み
ものみの塔聖書冊子協会,ものみの塔聖書冊子協会|ものみの塔聖書冊子協会(フジタカ連帯),,,タノムで2表記
```

**名寄せ優先度**
1. まず `得意先コード`（販売大臣）を基準に同定
2. 次に店舗名の文字列マッチング（完全一致→部分一致→類似）
3. 判断できない場合は手動で確認・追記

## 異常検知ルール

以下は**異常ではない**ので警告・除外しない:
- 金額が0円の行
- 水曜・日曜のデータが存在しない
- タノムにマイナス金額の行がある（返品処理）

以下は**要確認**として記録する:
- 同一店舗・同一日・同一金額の行が同ソース内に2件以上ある（重複の疑い）
- 想定規模から±30%以上の乖離（前週比で異常な増減）
- 販売大臣で `種` が 1/3/18 以外の値が出現した場合
